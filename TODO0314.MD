# TODO 0314 - Start Pray 前台 UX / UI 改善計畫

## 目標

從使用者角度重新檢視目前網站在手機版與桌機版的體驗缺口，整理出可執行的改善方向，優先修正「看不懂、找不到、按了沒反應、流程不一致」這四類問題。

這份文件整合了：

- 實際檢視 `https://startpray.online/prayfor` 的桌機版與手機版畫面
- 目前前端結構與互動邏輯的掃描
- 產品定位、使用者動機與首頁訊息策略

## 1. 產品定位與使用者動機

這一段要先釐清，因為如果平台不能回答「我為什麼要用你，而不是找朋友、教會、小組、LINE 群組、Facebook 社團就好」，使用者就不會留下來。

### 核心前提

使用者不是因為「想用一個禱告平台」才來，而是因為他當下有一個尚未被滿足的需求。

真正會驅動他進來的，不是功能名稱，而是以下幾種動機。

### 動機 1：我現在很需要被陪伴，但身邊沒有合適的人

**使用者心境**

- 我很想禱告，但我現在很孤單。
- 我不一定想公開發到朋友圈，也不一定有熟悉的屬靈同伴可以聊。
- 我希望有人願意聽我說，而不是只按一個讚。

**平台價值**

- 提供一個比社群貼文更安全、比聊天室更聚焦的空間。
- 讓使用者能快速說出需求，並得到真實回應。

**你要強調的訊息**

- 你不是一個人。
- 你的聲音會被聽見。
- 有人願意為你停下來。

### 動機 2：我不知道怎麼開口禱告，但我真的需要幫助

**使用者心境**

- 我不一定是成熟基督徒，甚至可能剛接觸信仰。
- 我遇到壓力、疾病、家庭問題、關係問題，但不知道怎麼表達。
- 我希望有一個入口，不用先很懂教會語言也能開始。

**平台價值**

- 降低進入門檻，讓「還不會禱告的人」也能開始。
- 讓錄音、文字、範例引導代替高門檻宗教術語。

**你要強調的訊息**

- 不會禱告也沒關係。
- 你可以先說出你的狀態。
- 平台會陪你走第一步。

### 動機 3：我願意為別人代禱，但我平常不知道去哪裡找到真實需要

**使用者心境**

- 我想服事、想關心別人，但平常沒有那麼多實際代禱需求進到我眼前。
- 我希望接觸的是活生生的人，而不是抽象口號。
- 我想知道自己的代禱與回應是有意義的。

**平台價值**

- 提供一個持續更新的代禱牆，讓願意付出的人更容易找到需要。
- 讓回應不只是閱讀，也可以是聲音、鼓勵、陪伴。

**你要強調的訊息**

- 你的代禱可以很具體。
- 你現在就能幫助一個人。
- 你的回應不是丟進空氣裡。

### 動機 4：我希望我的需要能被聽見，但我又在意隱私與安全

**使用者心境**

- 我有真實需要，但我不想暴露真名、真實身份或太多個資。
- 我怕被獵奇、被批評、被忽略。
- 我希望平台有基本的秩序與保護。

**平台價值**

- 提供匿名、暱稱、有限公開與內容管理機制。
- 讓求助者在可控制風險下被看見。

**你要強調的訊息**

- 你可以先用安全的方式被聽見。
- 平台不是完全失控的公開場域。
- 你的脆弱值得被尊重。

### 動機 5：我想用「聲音」而不是冰冷文字來彼此陪伴

**使用者心境**

- 純文字回覆很快，但有時候沒有溫度。
- 當我處在低潮、疾病、焦慮或重大壓力中，聲音比文字更有力量。
- 我想感受到對方是真的停下來回應我，而不是制式留言。

**平台價值**

- 錄音回應是你跟一般留言板、論壇、社團最大的差異化。
- 這也是最值得被放大的產品核心，不是附屬功能。

**你要強調的訊息**

- 這裡不只是被閱讀，而是被聽見。
- 一段聲音，可能比十句文字更能陪伴人。
- 這是平台最有辨識度的價值。

### 動機 6：我想持續關心特定主題，而不是一次性滑過

**使用者心境**

- 我對家庭、醫治、工作、福音、某個社會議題有長期負擔。
- 我不想只是看一篇內容就結束，我想追蹤後續。
- 我想知道被代禱的人後來怎麼樣了。

**平台價值**

- 若能建立追蹤、收藏、通知、更新，平台就不只是單次互動，而是陪伴關係。

**你要強調的訊息**

- 你不是看完就離開。
- 你可以持續同行。
- 每個代禱事項都可能有後續故事。

### 使用者應該如何理解這個產品

如果用使用者語言，而不是產品語言，這個平台比較像：

> 一個讓人可以安全說出需要、被別人用聲音真實陪伴的代禱平台。

不是：

- 一個很炫的錄音網站
- 一個代幣平台
- 一個宗教版社群媒體
- 一個很多功能的會員系統

代幣、會員中心、後台、分類、獎勵，這些都只是支撐系統。真正讓人留下來的，是：

- 我在這裡真的能被看見
- 我在這裡真的能幫助別人
- 我在這裡感受到陪伴是真的

## 2. 目前最明顯的體驗問題

### A. 導覽與資訊架構不一致

**觀察**

- 根路由 `/` 直接導到 `/prayfor`，但主選單仍顯示「首頁」，使用者容易以為會看到獨立首頁。
- Header 中未登入時隱藏「會員中心」，登入後才顯示，會讓新用戶難以理解登入後能做什麼。
- 頁面文案同時出現 `Start Pray`、`Let's Pray`、`Prayer Coin`，品牌識別不一致。

**使用者感受**

- 不確定自己目前在哪一頁。
- 不容易理解網站的主要流程與核心價值。
- 對平台信任感下降，像是還在開發中的產品。

**改善計畫**

- 統一品牌名稱、站點標題、頁尾文案、信件文案與按鈕文案。
- 決定 `/` 是否保留成真正 landing page；若保留 redirect，就把導覽文案改為「禱告牆」而非「首頁」。
- 在導覽中保留「會員中心」入口，但未登入時導向登入頁並清楚說明用途。

**優先級**

- P0

### B. 首頁敘事很多，但主流程不夠聚焦

**觀察**

- `/prayfor` 首屏資訊量很大，Hero 之後先出現介紹卡、統計卡，再進到搜尋與卡片牆，主要任務出現得偏晚。
- 首頁同時對信徒、非信徒、學習禱告者、想集氣者說話，但主要任務沒有被清楚排序。
- Hero 之後馬上接多段內容模組，使用者很難快速決定下一步。
- 首頁 CTA 有些會導向舊頁面或不同敘事方向，例如 `/legacy/search`。

**使用者感受**

- 看了很多內容，但不知道應該先「註冊」、「搜尋」、「發布需求」還是「先聽別人的故事」。
- 在手機上首屏後要滑很久，決策成本高。
- 首屏雖然有氛圍，但不像一個能立刻開始行動的入口。

**改善計畫**

- 首頁重新聚焦成三條主流程：
  1. 我想找人為我代禱
  2. 我想為別人代禱
  3. 我想先看看平台是怎麼運作
- 首屏保留單一主 CTA 與一個次 CTA，不要超過兩個。
- Hero 高度縮短，讓首屏可直接看見搜尋區或第一排卡片。
- 將搜尋、熱門牆提到平台介紹與統計前面。
- 將次要敘事區塊合併，減少重複的價值宣告。
- 移除或替換導向舊資源的 CTA，例如 `/legacy/search`。

**優先級**

- P0

### C. 手機版導覽與版頭雖可用，但仍有不穩定感

**觀察**

- `globals.css` 中有多段 header / mobile override，代表樣式是疊加修補出來的。
- 手機選單依賴絕對定位與多層 media query 覆蓋，後續很容易出現位移或遮蓋問題。
- 固定 header、固定播放器、固定左右切換箭頭會在小螢幕同時競爭空間。

**使用者感受**

- 手機上會有「畫面很滿、按鈕很多、內容被浮動元件擋住」的壓迫感。
- 使用者不一定知道目前打開的是選單、播放器還是頁面導覽功能。

**改善計畫**

- 將 mobile header 規則收斂成單一版本，移除重複 override。
- 明確定義手機版資訊層級：Header > 內容 > 播放器，不再額外常駐左右箭頭。
- 詳情頁在手機版把前後切換改成內容底部「上一篇 / 下一篇」卡片，不使用浮動按鈕。

**優先級**

- P1

### D. 搜尋與分類體驗可用，但互動節奏偏慢

**觀察**

- 搜尋 debounce 設為 2000ms，對使用者來說回饋太慢。
- 分類卡、搜尋建議、熱門牆是三套入口，但缺少明確預設路徑。
- 搜尋結果與分類切換時，缺少更清楚的空狀態、建議下一步與回復預設列表操作。
- 卡片目前主要資訊偏少，使用者較難快速判斷哪一則值得點進去。

**使用者感受**

- 輸入後會以為搜尋沒有反應。
- 搜尋沒結果時，不知道該改什麼關鍵字。
- 卡片很多，但不容易快速掃描與比較。

**改善計畫**

- 將搜尋 debounce 調整到 300ms 到 500ms。
- 在搜尋框下方加入「熱門關鍵字」與「找不到？改看熱門禱告」。
- 搜尋無結果時提供推薦分類與返回熱門牆按鈕。
- 強化卡片上的主要資訊層級：標題、類別、回應數、最近更新。
- 若資料足夠，補一行摘要或狀態 badge，例如「最新回應」「急需代禱」。

**優先級**

- P1

### E. 禱告詳情頁視覺強，但核心任務仍有落差

**觀察**

- 詳情頁有「關注」按鈕，但目前看起來沒有實作狀態回饋。
- 主要內容、留言、播放器、上下篇切換都同時存在，操作元素偏多。
- 手機版底部固定播放器 + 浮動切換箭頭，容易互相干擾。

**使用者感受**

- 不確定「關注」是否真的生效。
- 不知道這頁最重要的任務是閱讀、留言、播放錄音還是切換下一篇。
- 在手機上會感受到頁面被多個固定元件切碎。

**改善計畫**

- 移除未實作的假按鈕，或補上完整互動與成功狀態。
- 詳情頁重新定義主 CTA：
  - 第一優先：播放禱告 / 聽回應
  - 第二優先：留下回應
  - 第三優先：延伸瀏覽其他代禱事項
- 手機版把播放器縮成更低干擾的 mini player，並保留清楚的展開入口。
- 強化留言區表單提示，讓使用者知道可以文字回應或錄音回應。

**優先級**

- P0

### F. 登入、註冊、忘記密碼流程存在訊息矛盾

**觀察**

- 登入頁仍寫著「忘記密碼功能暫停中」，但系統已有 reset password API。
- `/forgot-password` 目前直接 redirect 到 `/login`，等於使用者看不到重設流程。
- 註冊表單欄位較多，對手機使用者負擔高。
- 條款文字寫成「免責聲、白皮書」，文案不完整。

**使用者感受**

- 覺得產品流程沒有整理好。
- 忘記密碼時會直接卡住。
- 第一次註冊時，容易中途放棄。

**改善計畫**

- 立即恢復真正的 `/forgot-password` 頁面，並從登入頁放出清楚入口。
- 將登入頁提示改成真實狀態，不使用過期文案。
- 註冊流程拆成兩步或精簡必填欄位，先完成帳號建立，再補齊個人資料。
- 條款與白皮書連結文案、資訊架構、視覺層級同步整理。

**優先級**

- P0

### G. 會員中心資訊很多，但手機版閱讀負擔偏重

**觀察**

- `customer-portal` 單頁承載個人資料、卡片列表、回應紀錄、編輯動作與彈窗。
- 桌機版資訊完整，但手機上容易變成超長頁面。
- 卡片與回應模組較重，缺少折疊、篩選、摘要統計。

**使用者感受**

- 想找自己的資料、貼文、回應時，需要滑很久。
- 手機上不容易快速完成「編輯個人資料」或「查看最新互動」。

**改善計畫**

- 會員中心改成分頁式架構：
  - 總覽
  - 我的禱告
  - 我的回應
  - 個人資料
- 手機版優先顯示摘要與最近活動，其餘區塊收合。
- 重要 CTA 固定在各分頁首屏，例如「新增禱告」「編輯個人資料」。

**優先級**

- P1

### H. 文案與信任訊號還不夠成熟

**觀察**

- 多處文案仍像內部開發語氣，例如「管理中心」「代幣動態」「成功率 99.8%」這類訊息若沒有佐證，容易降低信任。
- 有些頁面提到的功能尚未完整上線，例如通知、MFA、陌生裝置登入提醒。
- 頁尾聯絡資訊與社群入口還不完整。

**使用者感受**

- 產品像在展示 roadmap，而不是一個穩定可用的平台。
- 對資料安全、帳號可靠性、平台真實性會保留疑慮。

**改善計畫**

- 移除尚未實作的安全承諾與功能描述。
- 建立「目前可用功能」與「即將推出功能」的清楚邊界。
- 補上真實且一致的聯絡方式、FAQ、平台運作說明與隱私/使用條款。

**優先級**

- P0

### I. 視覺系統仍有拼接痕跡

**觀察**

- 全站同時存在多套視覺語言：淺色首頁、深色詳情頁、深色會員中心、舊版 legacy 資產。
- 有些圖片與資源仍來自 `/legacy`。
- `globals.css` 過大，且有重複 header / mobile / card 規則。

**使用者感受**

- 從列表頁切到詳情頁或會員中心時，像進到不同產品。
- 整體品牌感與專業度被削弱。

**改善計畫**

- 建立最小設計系統：
  - 色彩
  - 字級
  - 按鈕
  - 卡片
  - 表單
  - 狀態訊息
- 清理 legacy 圖片與舊路徑依賴。
- 把全域樣式拆成 `layout / auth / marketing / explorer / detail / portal` 幾個責任明確的檔案。

**優先級**

- P1

### J. 無障礙與可理解性還可再提升

**觀察**

- 目前已有一些 `aria-label` 與 `sr-only`，但互動較多的浮動元件仍可能造成閱讀器與鍵盤操作負擔。
- 搜尋建議、播放器、浮動導覽、卡片覆蓋連結都需要再次驗證 tab 順序與焦點狀態。
- 手機與長頁面操作時，焦點管理和狀態提示不足。

**使用者感受**

- 對鍵盤使用者與部分輔助工具使用者不夠友善。
- 某些動作完成後，沒有明確成功或失敗訊號。

**改善計畫**

- 針對 header menu、搜尋建議、播放器、留言表單進行 keyboard flow 檢查。
- 補齊 `focus-visible`、`aria-expanded`、`aria-current`、toast/alert 狀態。
- 對手機版固定元件進行可操作區域檢查，避免遮擋主要內容。

## 3. 從頁面層級補充的 UX 結論

這部分是從桌機版與手機版實際畫面補充出的重點，作為上面章節的橫向摘要。

### 桌機版

- 首屏視覺有氛圍，但主要任務出現偏慢。
- 搜尋與祈禱卡片牆其實是核心價值，卻被介紹卡與統計卡往後推。

### 手機版

- 首屏之後要滑過數個深色模組才會進入完整卡片牆，任務密度偏高。
- 字級、留白、浮動播放器共同壓縮了可視內容高度。
- 多個固定元件同時存在時，會讓使用者覺得畫面一直被 UI 擋住。

### 共通問題

- 主任務沒有被強化成最短路徑。
- 卡片資訊不足以支撐快速判斷。
- 語音回應這個差異化價值，雖然已存在，但還沒有被放到足夠核心的位置。

## 7. 首頁與文案策略建議

首頁真正應該回答的，不是「我們想成為什麼」，而是下面這幾句：

- 你現在有需要被代禱的事嗎？
- 你願意用一句話或一段聲音，把它說出來嗎？
- 你也可以現在就去陪伴另一個正在低谷中的人。

### 如果我是第一次來的使用者，我會留下來，如果我在 10 秒內看懂這三件事

1. 這裡是做什麼的  
   不是抽象地說願景，而是直接告訴我：你可以發出代禱需求，也可以為別人錄音回應。

2. 這裡跟別的平台有什麼不同  
   最大差異不是「也可以留言」，而是「你會收到真實的聲音回應」。

3. 我現在第一步應該做什麼  
   我要先搜尋、先聽、先註冊、還是先發佈需求？這件事必須非常清楚。

### 如果我是第一次來的使用者，我可能會離開的原因

- 我看不懂這個平台到底是禱告牆、社群網站、代幣平台，還是內容平台。
- 我看到太多段落，但不知道先做哪個動作。
- 我想按忘記密碼，結果功能不一致。
- 我看到一些像是還沒做完的功能，信任感下降。
- 我在手機上被太多固定按鈕、播放器、浮動元件干擾。
- 我不確定發出去的內容到底會不會有人回應。

## 9. `/admin` 後台補充掃描（2026-03-14）

以下是這次新增的後台掃描重點，已回填到上面的 P0/P1/P2 待辦。

### A. 權限與安全模型（P0）

**觀察**

- `/admin` 頁面可見性主要靠前端 `sessionStorage` 狀態控制。
- 登入頁 2FA 目前是前端硬編碼固定值，不是伺服器驗證。
- 多個 `/api/admin/*` 路由沒有伺服器端權限檢查，或只檢查 `x-admin-role` header。
- `x-admin-role` 屬於可由前端請求任意設定的 header，不能作為最終授權依據。

**風險**

- 權限可被繞過或偽造，屬於後台高風險漏洞。
- 後台資料讀寫（封鎖、設定、交易狀態）缺乏可信的 server-side gate。

**改善計畫**

- 建立伺服器端 session（httpOnly cookie + 簽章/驗證）並在所有 admin API 驗證。
- 將角色校驗集中化（例如 `requireAdmin("SUPER")`），不要分散在各路由。
- 補上登入防護：2FA 真實驗證、登入失敗節流、操作審計一致化。

### B. 後台 IA 與功能真實性（P0-P1）

**觀察**

- 存在已建立路由但未進入主導覽的後台頁（analytics、finance、marketing、moderation、support、home-categories）。
- 部分頁面是示意資料（硬編碼陣列），與真實管理流程混用。
- `/admin/content` 與 `/admin/marketing` 都能編輯 Banner，存在雙入口與資料來源分岔風險。

**使用者感受（管理員）**

- 看起來功能很多，但實際流程不連續。
- 團隊難以判斷哪些頁面可正式操作、哪些是規劃中功能。

**改善計畫**

- 將後台分成「正式可用」與「規劃中」兩層，導覽只留可用流程。
- 合併重疊模組（Banner 只留一個管理入口）。
- 對尚未上線頁面顯示明確標記，避免誤用。

### C. 手機版後台可用性（P1）

**觀察**

- 側欄在小螢幕使用 `transform: translateX(-100%)` 隱藏，但版型缺少可切換 `.is-open` 的實際控制。
- 多數管理頁以寬表格為主，缺少小螢幕下的橫向捲動容器或卡片替代。
- 篩選列 + 分頁 + 操作按鈕同時存在時，手機版資訊壓力高。

**改善計畫**

- 補上 mobile sidebar toggle 與明確收合行為。
- 為表格提供行動版策略：水平卷軸、關鍵欄位優先、次要欄位折疊。
- 把高頻操作固定在可見區，降低長表格中的來回滾動成本。

### D. 操作回饋與一致性（P1-P2）

**觀察**

- 後台同時使用 `alert`、`confirm`、頁內錯誤文字，回饋風格不一致。
- 狀態文案中英混用（Active/Blocked/中文），不同頁呈現不一致。
- 重要操作（封鎖、交易狀態更新）缺乏一致的確認與成功回饋模式。

**改善計畫**

- 統一後台 feedback system：toast + inline error + confirm dialog。
- 統一狀態詞與文案語氣，建立 admin copy style。
- 對關鍵操作提供可追溯的成功訊息（誰、何時、改了什麼）。

### E. 建議先做的 `/admin` 三件事

1. 先補齊伺服器端授權與 session 驗證（P0）。
2. 清理示意頁與重複入口，讓後台 IA 只保留可用流程（P0-P1）。
3. 修復手機側欄與表格行動版可用性（P1）。
